# 加密货币合约监控告警系统重构方案

## 1. 文档目的

本文档用于整理当前监控告警项目的重构思路，给出一套适合后续演进的目标架构、模块边界、数据模型、组件选型与迁移路径。

本文档的目标不是立即约束所有实现细节，而是回答以下问题：

- 现有架构的主要问题是什么
- 目标架构应如何拆分职责
- 数据库和消息中间件分别承担什么角色
- 是否应该取消当前内存缓冲区
- 各个脚本如何实现“低耦合、高可替换”
- 建议优先落地哪些改造步骤

---

## 2. 背景与现状

当前项目采用单进程、多模块协作的方式运行，观察列表扫描、规则检测、最近告警缓存、Web 面板、外部推送等能力运行在同一套进程生命周期中。

这一模式在早期开发阶段具备实现简单、上线快、排障直接等优点，但随着需求增加，逐渐暴露出以下问题：

1. 观察列表筛选、规则检测、推送发送、Web 展示运行耦合较深。
2. 告警最近记录依赖内存缓冲区，重启即丢失，无法完整审计。
3. Web 面板展示与告警处理共享内存状态，边界不清晰。
4. 推送是否成功、推送失败原因、重试情况缺乏独立持久化记录。
5. 后续若继续扩展更多规则、更多通道、更多数据源，维护成本会明显上升。

因此，需要将系统从“进程内共享状态的轻量实现”演进为“以数据契约为边界的解耦架构”。

---

## 3. 重构目标

### 3.1 核心目标

1. 将观察列表筛选、规则检测、推送执行、Web 查询分离为独立职责模块。
2. 用数据库替代当前进程内关键状态，确保状态可追溯、可恢复、可审计。
3. 用消息中间件承接异步事件流，降低模块间的直接调用关系。
4. 取消当前 `RecentAlertsBuffer` 这类纯内存缓冲区设计。
5. 为后续新增规则、新增通道、新增数据源预留扩展空间。

### 3.2 非目标

本次重构方案不建议一开始就追求以下目标：

- 不建议一步到位拆成大量重型微服务
- 不建议立即引入过多基础设施组件
- 不建议为了“完全无关系”而让各模块缺乏统一数据规范

目标应是：

**通过数据库表结构、事件结构、主键规范、时间语义，实现低耦合，而不是通过“文件之间互不认识”实现伪解耦。**

---

## 4. 设计原则

### 4.1 数据契约优先

各模块之间不直接共享进程内对象，不依赖内存缓存，不依赖隐式状态，而是通过以下边界协作：

- 数据库表
- 消息队列事件
- 只读查询接口

### 4.2 职责单一

每个模块只做一类事情：

- 采集模块只负责采集与标准化
- 筛选模块只负责观察列表计算
- 规则模块只负责判定是否命中
- 推送模块只负责发送与重试
- Web 端只负责查询展示

### 4.3 状态持久化

所有关键业务状态应可在数据库中查询和恢复，包括：

- 当前观察列表
- 观察列表变更历史
- 告警事件
- 推送尝试记录
- 推送结果

### 4.4 渐进式演进

优先选择“从现有代码平滑迁移”的方案，避免一次性大爆炸改造。

---

## 5. 目标架构概览

推荐采用：

**PostgreSQL + 轻量消息中间件 + 独立 Worker + 只读查询 API**

### 5.1 总体说明

系统拆分为以下几类角色：

1. 市场数据采集模块
2. 观察列表筛选模块
3. 规则检测模块
4. 告警富化模块
5. 推送执行模块
6. 查询接口模块
7. Web 展示模块

数据库负责存储事实数据与状态快照。  
消息中间件负责在模块之间传递异步事件。  
Web 面板不再读取进程内缓冲区，而是只读数据库或只读查询 API。

### 5.2 逻辑流转

```text
交易所 / 外部数据源
        ↓
市场采集模块
        ↓
数据库（行情快照 / K线 / OI）
        ↓
观察列表筛选模块 ──→ watchlist.updated 事件
        ↓
数据库（watchlist_snapshot / watchlist_history）
        ↓
规则检测模块 ──→ alert.created 事件
        ↓
数据库（alert_event）
        ↓
告警富化 / 推送模块 ──→ delivery.result 事件
        ↓
数据库（delivery_attempt）
        ↓
查询 API
        ↓
Web 面板 / 外部管理端
```

## 6. 模块划分建议

### 6.1 `market-ingestor`

职责：

- 从 Binance 或其他交易所采集价格、K 线、OI、交易所元数据
- 对外部数据进行标准化
- 落库到行情相关表

不负责：

- 不负责筛选观察列表
- 不负责触发告警
- 不负责发送通知

### 6.2 `watchlist-selector`

职责：

- 基于数据库中的市场快照执行观察列表筛选
- 生成当前观察列表结果
- 写入观察列表快照与历史表
- 发布 `watchlist.updated` 事件

不负责：

- 不负责推送
- 不负责面板展示

### 6.3 `rule-engine`

职责：

- 读取当前观察列表与行情数据
- 执行规则计算
- 生成结构化告警事件
- 写入 `alert_event`
- 发布 `alert.created`

不负责：

- 不负责具体推送通道实现
- 不负责 Web 展示

### 6.4 `alert-enricher`（可选独立，也可并入推送模块）

职责：

- 补充市值、OI 价值、标签、模板字段
- 将可展示/可推送的上下文补齐

说明：

如果系统规模不大，可以先不单独拆服务，作为推送前的一个处理阶段即可。

### 6.5 `notification-worker`

职责：

- 消费 `alert.created`
- 根据规则、通道配置执行推送
- 记录每次推送尝试
- 处理失败重试、限流和错误日志

输出：

- `delivery_attempt`
- `delivery.result`

### 6.6 `query-api`

职责：

- 为 Web 面板提供只读查询接口
- 聚合数据库中的观察列表、告警事件、推送结果
- 提供审计与回溯能力

不负责：

- 不直接参与告警计算
- 不直接维护内存态缓存

---

## 7. 数据库设计建议

推荐主数据库使用 `PostgreSQL`。

原因：

- 关系模型清晰
- 事务与一致性能力较好
- 适合存储告警、快照、发送记录等结构化数据
- 后续可扩展分区、物化视图、Timescale 等能力

### 7.1 核心表

#### 7.1.1 `instrument_snapshot`

用途：

- 存储最新标的行情快照

建议字段：

- `symbol`
- `snapshot_time`
- `mark_price`
- `open_interest_qty`
- `open_interest_value`
- `volume_1m`
- `volume_5m`
- `source`

#### 7.1.2 `watchlist_snapshot`

用途：

- 存储当前时刻观察列表中的标的状态

建议字段：

- `symbol`
- `is_active`
- `entry_time`
- `reason_codes`
- `reason_texts`
- `rank_oi7d`
- `entry_price`
- `max_price_since_entry`
- `max_oi_since_entry`
- `updated_at`

#### 7.1.3 `watchlist_reason_history`

用途：

- 记录观察列表的原因变化历史，便于回溯

建议字段：

- `id`
- `symbol`
- `reason_code`
- `action`（added / removed / refreshed）
- `effective_time`
- `source_job_id`

#### 7.1.4 `alert_event`

用途：

- 存储规则命中的事实事件

建议字段：

- `event_id`
- `symbol`
- `rule_type`
- `triggered_at`
- `watchlist_entry_at`
- `watchlist_reasons`
- `payload_json`
- `dedup_key`
- `created_at`

#### 7.1.5 `delivery_attempt`

用途：

- 存储每次推送尝试与结果

建议字段：

- `id`
- `event_id`
- `channel`
- `target`
- `attempt_no`
- `status`（pending / success / failed / skipped）
- `error_message`
- `response_code`
- `response_body`
- `started_at`
- `finished_at`

### 7.2 关键原则

1. `alert_event` 与 `delivery_attempt` 必须分开。
2. “是否触发告警”与“是否成功发送”是两类不同事实。
3. “当前状态表”与“历史审计表”必须分开。
4. 所有事件都需要稳定主键，避免重复发送与重复入库。

## 8. 消息中间件设计建议

### 8.1 作用

消息中间件用于承接以下场景：

- 观察列表更新通知
- 新告警事件通知
- 推送结果回传

其作用是让各模块不需要直接函数调用，也不需要运行在同一进程中。

### 8.2 推荐选型

#### 第一优先级

- `RabbitMQ`
- `Redis Streams`

适用原因：

- 上手成本低
- 运维复杂度相对可控
- 足以应对当前这类中等吞吐、以任务分发和异步解耦为主的场景

#### 后续可升级

- `Kafka`

适用前提：

- 事件量明显增大
- 需要长期保留事件流
- 需要多消费者回放历史事件

### 8.3 推荐主题/队列

- `watchlist.updated`
- `alert.created`
- `delivery.result`

如果初期复杂度需要再降，可以先不把所有事件都上消息队列，而是采用：

**数据库落库 + 关键动作发消息**

---

## 9. 关于“取消缓冲区”的建议

当前 `RecentAlertsBuffer` 是进程内内存缓冲区，仅适用于轻量展示。

建议取消，原因如下：

1. 重启丢失
2. 无法审计
3. 与 Web 面板逻辑耦合
4. 与推送成功/失败的事实不一致

替代方案：

- 用 `alert_event` 作为“近期告警”的事实来源
- 用 `delivery_attempt` 作为“推送情况”的事实来源
- Web 面板通过查询 API 读取数据库中的最近事件

### 9.1 替代后的收益

- 最近告警可追溯
- 推送状态可区分成功、失败、跳过
- 页面重启、服务重启后数据仍保留
- 声音提示、列表展示、推送状态都可基于同一事实源

---

## 10. “各个脚本之间无关联”的正确理解

这条需求建议改写为：

**各模块之间不做进程内强耦合，不共享内存状态，只通过数据库与消息事件协作。**

不建议理解为：

- 完全没有统一字段
- 完全没有统一主键
- 完全没有统一时间标准
- 每个模块自由定义自己的数据结构

如果按“毫无关系”字面执行，结果通常会更糟：

- 代码耦合下降了
- 数据耦合反而失控
- 最终表结构混乱、事件格式不一致、排障更困难

因此应坚持：

### 10.1 统一的数据契约

至少统一以下内容：

- `symbol` 命名规范
- `event_id` 生成方式
- 时间字段统一时区与精度
- 规则类型枚举
- 推送通道枚举
- 状态字段枚举

### 10.2 松耦合实现方式

- 不直接调用对方模块内部函数
- 不依赖对方进程内状态
- 不读取对方临时内存缓存
- 只认数据库和事件消息

## 11. 推荐技术选型

### 11.1 推荐组合

对于当前阶段，推荐优先采用：

- 主数据库：`PostgreSQL`
- 消息中间件：`RabbitMQ` 或 `Redis Streams`
- 查询 API：`FastAPI`
- Worker：Python 独立进程
- 调度：`cron` / `APScheduler`

### 11.2 不建议立即采用的重型方案

以下方案可以作为未来选项，但不建议当前立刻引入：

- 全量微服务化
- Kafka 全链路
- 多数据库混用
- 复杂服务编排

原因：

- 当前改造重点是“拆清边界、统一模型、持久化关键状态”
- 不是“堆叠尽可能多的基础设施”

---

## 12. 迁移路线建议

建议分阶段推进。

### Phase 1：先落库

目标：

- 引入 PostgreSQL
- 将观察列表、告警事件、推送尝试落库
- 保留当前主流程，但让关键状态先持久化

这一步完成后，系统已经能摆脱“重启即丢关键状态”的问题。

### Phase 2：拆推送与查询

目标：

- 独立 `notification-worker`
- 独立 `query-api`
- Web 面板从数据库读“最近告警”和“推送状态”

这一步完成后，当前内存缓冲区可以被替代。

### Phase 3：拆筛选与规则

目标：

- 将观察列表筛选与规则检测拆成独立进程
- 通过数据库 + 队列协作

这一步完成后，主要业务职责就已经解耦。

### Phase 4：视规模再上更复杂组件

目标：

- 评估是否引入 Kafka
- 评估是否引入时序库
- 评估是否做多消费者回放与重放能力

前提：

- 确实出现性能瓶颈、吞吐增长或审计回放要求

---

## 13. 推荐最终形态

对于当前项目，最合适的中期目标不是“极重微服务”，而是：

**模块化单体向事件驱动架构逐步演进**

即：

1. 先统一数据模型
2. 先持久化关键状态
3. 再拆 Worker 和 API
4. 再拆筛选和规则模块
5. 最后按规模决定是否继续微服务化

这是当前阶段性价比最高、风险最低、可演进性最强的方案。

---

## 14. 结论

当前规划总体合理，尤其是以下方向是正确的：

- 使用数据库替代关键内存状态
- 使用消息中间件进行异步解耦
- 拆分观察列表、规则检测、推送执行、查询展示
- 取消最近告警内存缓冲区

但需要修正的点在于：

- 不建议追求字面意义上的“脚本之间毫无关系”
- 应改为“模块之间通过统一数据契约协作”
- 应优先做“数据模型和边界治理”，而不是优先做“服务数量增加”

最终推荐方案为：

**PostgreSQL + RabbitMQ/Redis Streams + FastAPI + 独立 notification worker**

在此基础上逐步演进，而不是一次性推翻重来。
