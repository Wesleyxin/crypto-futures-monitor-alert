# 监控告警系统部署文档

## 1. 文档目的

本文档描述基于目标重构架构的监控告警系统部署方案，覆盖以下内容：

- 部署目标与适用范围
- 推荐的环境规划与组件选型
- 单机版与分层版部署拓扑
- 目录规范、配置规范、端口规划
- 详细部署步骤
- 上线检查、监控、备份、升级与回滚建议

本文档对应的是**目标架构**，不是当前仓库中“单进程 + 内存缓冲区 + 内置 Web 面板”的旧部署方式。  
当前仓库已有的轻量部署方法仍可参考 `README.md`；本文档用于后续重构后的正式部署设计。

---

## 2. 适用范围

本文档适用于以下目标系统：

- 使用 `PostgreSQL` 作为主数据库
- 使用 `RabbitMQ` 或 `Redis Streams` 作为消息中间件
- 将系统拆分为独立职责模块
- 通过 `query-api` 向 Web 面板提供只读查询
- 不再依赖进程内 `RecentAlertsBuffer`

推荐部署的模块包括：

1. `market-ingestor`
2. `watchlist-selector`
3. `rule-engine`
4. `notification-worker`
5. `query-api`
6. `postgres`
7. `rabbitmq` 或 `redis`
8. `nginx`（可选，用于反向代理与 TLS）

---

## 3. 部署原则

### 3.1 核心原则

1. **数据库优先落地**：关键状态必须先可持久化、可查询。
2. **服务职责单一**：一个服务只做一类事情。
3. **先简单后复杂**：优先单机部署跑通，再分层、再扩容。
4. **配置与代码分离**：敏感信息不写入代码仓库。
5. **部署可回滚**：每次上线都应能快速恢复到前一版本。

### 3.2 推荐部署策略

对于当前阶段，推荐采用：

**单机多容器 / 单机多服务部署**

原因：

- 成本低
- 运维简单
- 便于先跑通重构后的全链路
- 便于验证表结构、消息流、推送链路

当系统稳定后，再考虑拆分为：

- 数据层单独机器
- 业务 Worker 单独机器
- API / Web 单独机器

## 4. 目标部署拓扑

## 4.1 推荐拓扑 A：单机版（第一阶段）

适合：

- 当前阶段
- 中小规模
- 优先验证业务链路

```text
+-----------------------------------------------------------+
|                   Linux Server                            |
|                                                           |
|  +-------------------+    +---------------------------+   |
|  | PostgreSQL        |    | RabbitMQ / Redis Streams |   |
|  +-------------------+    +---------------------------+   |
|                                                           |
|  +-------------------+    +---------------------------+   |
|  | market-ingestor   |    | watchlist-selector        |   |
|  +-------------------+    +---------------------------+   |
|                                                           |
|  +-------------------+    +---------------------------+   |
|  | rule-engine       |    | notification-worker       |   |
|  +-------------------+    +---------------------------+   |
|                                                           |
|  +-------------------+    +---------------------------+   |
|  | query-api         |    | nginx / web panel         |   |
|  +-------------------+    +---------------------------+   |
|                                                           |
+-----------------------------------------------------------+
```

## 4.2 推荐拓扑 B：分层版（第二阶段）

适合：

- 业务稳定后
- 推送量增加
- 希望隔离数据库与业务负载

```text
+---------------------+     +---------------------+     +----------------------+
|  DB / MQ Server     |     |  Worker Server      |     |  API / Web Server    |
|---------------------|     |---------------------|     |----------------------|
| PostgreSQL          |     | market-ingestor     |     | query-api            |
| RabbitMQ / Redis    |     | watchlist-selector  |     | nginx                |
| backup agent        |     | rule-engine         |     | web panel            |
|                     |     | notification-worker |     |                      |
+---------------------+     +---------------------+     +----------------------+
```

## 5. 组件说明

| 组件 | 作用 | 是否必须 | 建议 |
|------|------|----------|------|
| `postgres` | 主数据库，存储观察列表、告警、推送记录 | 必须 | 第一阶段即上线 |
| `rabbitmq` / `redis` | 承接异步事件流 | 建议 | 第一阶段建议上线 |
| `market-ingestor` | 市场数据采集与标准化 | 必须 | 独立服务 |
| `watchlist-selector` | 生成观察列表快照 | 必须 | 独立服务 |
| `rule-engine` | 告警规则计算 | 必须 | 独立服务 |
| `notification-worker` | 执行推送与重试 | 必须 | 独立服务 |
| `query-api` | 向面板提供只读接口 | 必须 | 独立服务 |
| `nginx` | 反向代理、TLS、统一入口 | 可选但推荐 | 生产环境建议使用 |

## 6. 环境规划

## 6.1 环境划分

建议至少区分以下环境：

### `dev`

用途：

- 本地开发
- 联调
- 表结构验证

特点：

- 可与本地数据库共享
- 推送通道可接测试 Webhook

### `staging`

用途：

- 预发布验证
- 配置验证
- 部署流程演练

特点：

- 与生产相同部署结构
- 数据量可以较小
- 推送建议使用单独测试群或测试频道

### `prod`

用途：

- 正式生产运行

特点：

- 与 staging 保持一致的部署形态
- 使用正式 Webhook 与正式数据库备份策略

---

## 6.2 推荐资源规格

### 单机版推荐起步配置

- CPU：`4 vCPU`
- 内存：`8 GB`
- 磁盘：`100 GB SSD`

### 预估分配

| 组件 | CPU | 内存 |
|------|-----|------|
| PostgreSQL | 1~2 核 | 2~3 GB |
| RabbitMQ / Redis | 0.5~1 核 | 0.5~1 GB |
| market-ingestor | 0.5 核 | 0.5 GB |
| watchlist-selector | 0.5 核 | 0.5 GB |
| rule-engine | 0.5~1 核 | 0.5~1 GB |
| notification-worker | 0.5 核 | 0.5 GB |
| query-api + nginx | 0.5 核 | 0.5 GB |

如果未来要：

- 增加交易所
- 增加高频 K 线周期
- 增加更多告警规则
- 增加大量推送通道

则需要考虑拆分数据库与 Worker。

## 8. 网络与端口规划

推荐端口如下：

| 组件 | 端口 | 说明 |
|------|------|------|
| PostgreSQL | `5432` | 数据库 |
| RabbitMQ | `5672` | AMQP |
| RabbitMQ 管理台 | `15672` | 仅内网开放 |
| Redis | `6379` | 如使用 Redis Streams |
| query-api | `8080` | 内部 API |
| nginx / Web | `80` / `443` | 外部访问 |

建议：

1. PostgreSQL 只对内网开放。
2. RabbitMQ 管理台不要暴露到公网。
3. `query-api` 由 `nginx` 反向代理，不直接公网裸露。
4. Web 面板统一通过 `https://域名/` 访问。

## 9. 配置与密钥管理

## 9.1 配置分层

建议配置分为三类：

### 应用级配置

例如：

- 交易所 API 地址
- 轮询周期
- 规则阈值
- 是否启用某个模块

建议使用：

- `app.yaml`
- 或拆分为 `app.prod.yaml` / `app.staging.yaml`

### 环境变量配置

例如：

- 数据库连接串
- 消息中间件连接串
- 日志级别
- Web 服务监听地址

建议使用：

- `prod.env`
- `staging.env`

### 敏感密钥

例如：

- Discord Webhook
- 企业微信 Webhook
- API Token
- JWT Secret

建议：

- 不写入仓库
- 不写入示例配置中的真实值
- 通过 `.env`、密钥管理系统或 CI/CD Secret 注入

## 9.2 示例环境变量

WECOM_WEBHOOK_URL=https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx

QUERY_API_HOST=0.0.0.0

## 10. 推荐部署方式

推荐优先使用：

**Docker Compose + systemd 托管**

原因：

- 易于复制环境
- 易于启动多组件
- 便于区分服务边界
- 后续迁移到 Kubernetes 或其他编排方式也更自然

如果当前阶段不想上容器，也可以用：

- Python 虚拟环境 + 多个 `systemd` 服务

但从长期维护看，容器方式更适合目标架构。

## 11. 单机版部署步骤（推荐）

以下示例假设系统为：

- Ubuntu 22.04 / Debian 12 / CentOS Stream 9 之一
- 使用 Docker Engine 与 Docker Compose 插件

## 11.1 安装基础环境

### Ubuntu / Debian

```bash
sudo apt update
sudo apt install -y ca-certificates curl gnupg git
```

### CentOS / RHEL

```bash
sudo dnf install -y git curl ca-certificates
```

## 11.2 安装 Docker

请按你的发行版官方文档安装 Docker Engine 与 Compose 插件。  
安装完成后验证：

```bash
docker --version
docker compose version
```

## 11.3 创建目录

```bash
sudo mkdir -p /srv/crypto-futures-monitor/{app,config,data,logs,deploy}
sudo chown -R $USER:$USER /srv/crypto-futures-monitor
```

## 11.4 上传代码

```bash
cd /srv/crypto-futures-monitor
git clone <your-repo-url> app
```

如果不是 git 拉取，也可以直接上传代码包到 `app/`。

## 11.5 准备配置文件

建议至少准备以下文件：

```text
/srv/crypto-futures-monitor/config/prod.env
/srv/crypto-futures-monitor/config/app.yaml
/srv/crypto-futures-monitor/deploy/docker-compose.yml
```

## 11.6 准备持久化目录

```bash
mkdir -p /srv/crypto-futures-monitor/data/postgres
mkdir -p /srv/crypto-futures-monitor/data/rabbitmq
mkdir -p /srv/crypto-futures-monitor/data/backup
mkdir -p /srv/crypto-futures-monitor/logs/query-api
mkdir -p /srv/crypto-futures-monitor/logs/workers
```

## 11.7 启动基础设施

先启动数据库与消息中间件：

```bash
cd /srv/crypto-futures-monitor/deploy
docker compose up -d postgres rabbitmq
```

检查状态：

```bash
docker compose ps
docker compose logs -f postgres
docker compose logs -f rabbitmq
```

## 11.8 初始化数据库

建议使用以下方式之一：

1. 应用内置 migration 工具
2. `alembic`
3. 单独 SQL 初始化脚本

例如：

```bash
docker compose exec query-api python -m app.migrate
```

或者：

```bash
psql "$POSTGRES_DSN" -f /srv/crypto-futures-monitor/app/sql/init.sql
```

## 11.9 启动业务服务

```bash
docker compose up -d market-ingestor watchlist-selector rule-engine notification-worker query-api nginx
```

## 11.10 检查服务

```bash
docker compose ps
docker compose logs -f query-api
docker compose logs -f notification-worker
```

应重点确认：

- 数据库连接成功
- 队列连接成功
- 观察列表筛选能正常写库
- 规则检测能正常写入 `alert_event`
- 推送模块能正常记录 `delivery_attempt`
- `query-api` 健康检查正常

## 12. 推荐的 `docker-compose.yml` 结构

这里给出建议结构，不代表当前仓库已实现。

      - ../config/prod.env

    build: ../app

      - ../config/prod.env

    build: ../app

## 13. `systemd` 托管建议

如果采用 Docker Compose，推荐仍然通过 `systemd` 托管启动。

例如创建：

`/etc/systemd/system/crypto-futures-monitor.service`

```ini
[Unit]
Description=Crypto Futures Monitor Stack
After=network.target docker.service
Requires=docker.service

[Service]
Type=oneshot
WorkingDirectory=/srv/crypto-futures-monitor/deploy
ExecStart=/usr/bin/docker compose up -d
ExecStop=/usr/bin/docker compose down
RemainAfterExit=yes
TimeoutStartSec=0

[Install]
WantedBy=multi-user.target
```

启用：

```bash
sudo systemctl daemon-reload
sudo systemctl enable crypto-futures-monitor
sudo systemctl start crypto-futures-monitor
```

查看状态：

```bash
sudo systemctl status crypto-futures-monitor
```

## 14. 健康检查与上线验收

## 14.1 健康检查建议

每个服务建议提供健康检查：

| 服务 | 建议健康检查 |
|------|--------------|
| `postgres` | 连接测试 |
| `rabbitmq` | 连接测试 / 队列状态 |
| `market-ingestor` | 最近一次采集时间 |
| `watchlist-selector` | 最近一次成功筛选时间 |
| `rule-engine` | 最近一次成功规则计算时间 |
| `notification-worker` | 最近一次消费成功时间 |
| `query-api` | `/healthz` 或 `/api/health` |

## 14.2 上线验收清单

上线后至少检查：

1. 数据库表已正确创建。
2. 行情快照表有持续写入。
3. `watchlist_snapshot` 有数据。
4. 告警触发后 `alert_event` 有记录。
5. 推送后 `delivery_attempt` 有记录。
6. 推送成功和失败能够区分。
7. Web 面板能查到最近告警与推送状态。
8. 服务重启后状态不丢失。

## 15. 日志与监控

## 15.1 日志建议

每个服务独立日志，至少包含：

- 服务名
- 环境
- 时间
- 事件 ID
- 标的
- 规则类型
- 推送通道
- 错误原因

## 15.2 推荐监控项

### 业务监控

- 最近一次行情采集时间
- 最近一次观察列表刷新时间
- 最近一次规则计算时间
- 最近 5 分钟告警数
- 最近 5 分钟推送失败数

### 资源监控

- CPU
- 内存
- 磁盘
- PostgreSQL 连接数
- 队列积压长度

### 告警建议

至少配置以下运维告警：

- 数据库不可用
- 队列不可用
- 连续 N 分钟无行情写入
- 连续 N 分钟无观察列表更新
- 推送失败率异常升高

## 16. 备份与恢复

## 16.1 备份重点

必须备份：

- PostgreSQL 数据库
- 关键配置文件
- 环境变量模板（不含真实敏感值也要保留键）

## 16.2 推荐备份方式

### 数据库

每日：

- 全量逻辑备份一次

每小时：

- 增量或 WAL 级别备份（如果生产重要性较高）

### 配置文件

每次变更后：

- 备份 `prod.env`
- 备份 `app.yaml`
- 备份 `nginx.conf`

## 16.3 恢复演练

建议每月至少做一次恢复演练：

1. 从备份恢复 PostgreSQL
2. 恢复部署目录
3. 恢复配置文件
4. 启动服务
5. 校验 `watchlist_snapshot`、`alert_event`、`delivery_attempt`

---

## 17. 升级与回滚

## 17.1 升级流程建议

1. 在 `staging` 验证
2. 备份数据库
3. 备份配置
4. 拉取新版本代码
5. 执行 migration
6. 重启服务
7. 验证健康检查
8. 观察 10~30 分钟

## 17.2 回滚策略

如果升级失败：

1. 切回上一个镜像版本或代码版本
2. 如 migration 不兼容，需要回滚数据库变更
3. 恢复旧配置
4. 重启所有业务服务

注意：

- 数据库 schema 变更必须遵循可回滚原则
- 不建议直接做不可逆的破坏性 migration

## 18. 常见故障排查

## 18.1 页面打不开

排查顺序：

1. `nginx` 是否正常运行
2. `query-api` 是否正常运行
3. 反向代理配置是否正确
4. 防火墙 / 安全组是否放行

## 18.2 有告警但没推送

排查顺序：

1. `alert_event` 是否生成
2. `notification-worker` 是否消费到事件
3. `delivery_attempt` 是否记录失败
4. Webhook 是否失效
5. 外部网络是否可达

## 18.3 没有观察列表

排查顺序：

1. `market-ingestor` 是否正常写行情
2. `watchlist-selector` 是否正常执行
3. `watchlist_snapshot` 是否有数据
4. 配置阈值是否过高

## 18.4 队列堆积

排查顺序：

1. 是否有 worker 挂掉
2. 推送通道是否卡住
3. 单条消息处理是否超时
4. 是否需要增加 worker 副本

## 19. 当前阶段的落地建议

如果你接下来按重构路线推进，建议部署顺序如下：

### 第一阶段

先上线：

- `postgres`
- `query-api`
- `notification-worker`

目标：

- 先让告警与推送记录可持久化
- 先让 Web 查询脱离内存缓冲区

### 第二阶段

再拆：

- `watchlist-selector`
- `rule-engine`

目标：

- 完成观察列表与规则计算的服务解耦

### 第三阶段

最后评估：

- 是否要把数据采集进一步独立
- 是否需要 Kafka
- 是否需要时序数据库

---

## 20. 结论

对于当前项目，最推荐的部署路线是：

**单机多服务先跑通，数据库与消息中间件先稳定，再逐步拆分 Worker 与 API。**

不推荐一开始就：

- 上大量机器
- 上复杂微服务编排

**PostgreSQL + RabbitMQ/Redis Streams + query-api + notification-worker + watchlist-selector + rule-engine**
