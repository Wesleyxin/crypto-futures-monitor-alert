推荐主存储
核心原则： 不追求“脚本彼此完全无关系”，而是追求“模块之间不共享进程内状态，只共享数据库事实和事件契约”。

总体结构图

为什么这样拆

保留的能力
  • 观察列表仍然可以独立筛选，只是不再直接驱动页面内存状态。
  • 规则引擎仍然专注于命中逻辑，不承担外部推送失败的复杂度。
  • Web 面板仍然展示最近告警，但数据来源改为数据库事实表。
当前痛点对应解法
  • “响了但没看到新增”这类问题，会从前端内存逻辑变成数据库可追踪事实。
  • “告警触发”和“推送成功”拆成两张事实表，状态边界清晰。
  • 服务重启后状态不丢，排障不再依赖单次进程日志。

模块职责表

模块 输入 输出 不负责什么
market-ingestor 交易所 API 行情快照 / K线 / OI 不负责规则、不负责推送
watchlist-selector 行情快照 watchlist_snapshot + watchlist.updated 不负责页面、不负责通知
rule-engine 观察列表 + 行情 alert_event + alert.created 不负责通道发送
notification-worker alert.created delivery_attempt + delivery.result 不负责规则计算
query-api 数据库 面板查询结果 不直接消费交易所数据

核心存储设计

主数据库
  • watchlist_snapshot：当前观察列表状态
  • watchlist_reason_history：原因变化历史
  • alert_event：规则命中事实
  • delivery_attempt：每个通道每次发送尝试
  • instrument_snapshot：最近行情快照
消息中间件
  • watchlist.updated：观察列表变化事件
  • alert.created：新告警事件
  • delivery.result：推送结果事件
  • 初期建议 RabbitMQ 或 Redis Streams，不急着上 Kafka。

推荐技术选型

层 推荐 理由
主数据库 PostgreSQL 结构化强、事务可靠、适合当前体量
消息队列 RabbitMQ / Redis Streams 先低复杂度实现异步解耦
API FastAPI 管理接口和只读查询都适合
Worker Python 独立进程 与现有技术栈连续
调度 cron / APScheduler 先满足定时任务触发

迁移阶段图

阶段 做什么 结果
Phase 1 把观察列表、告警、推送记录先落库 先摆脱内存关键状态
Phase 2 拆 query-api 和 notification-worker 页面查询与推送执行解耦
Phase 3 拆 watchlist-selector 和 rule-engine 业务职责边界清晰
Phase 4 按体量评估 Kafka / 时序库 只在真正需要时升级基础设施

一句话结论：最推荐的落地路线不是一步到位重型微服务，而是 PostgreSQL + 轻量消息中间件 + 独立 worker + 只读 query-api 的渐进式事件驱动架构。
